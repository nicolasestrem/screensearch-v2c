# ScreenSearch V2c — 0.2.0 Roadmap: Attention-First Text Retrieval + Recall Reports

## Why
Today the app indexes **raw full-screen OCR with zero filtering** — the same text feeds both
full-text search and embeddings (`crates/store/src/schema.rs`: `ocr_text` → `ocr_text_fts`;
`crates/kernel/src/worker_pool.rs`: the whole frame's OCR is embedded as one chunk). So search,
Ask, and embeddings get dominated by chrome — taskbar, desktop icons, browser toolbars, even the
app's own sidebar labels. Searching "Firefox" / "Steam" / "Deck" surfaces frames purely because
those static labels were on screen.

Core rule for 0.2.0:

- Preserve raw full-screen text.
- Derive a default **content text** layer (filtered OCR/UIA text — *not* vision descriptions).
- Search, Ask, embeddings, and reports use content text by default.
- Users can opt into raw / app-chrome search when they need it. Default search stays hybrid
  (FTS over content text + vector); the FTS fallback is never removed.

## Scope

**Ships in 0.2.0** (build order): PR1 (specs) → PR2 (data model + OCR spans) → PR3 (attention-first
filtering) → PR6 (Reports + premade cards) → PR7 (audit).

**Deferred to 0.2.1** (separate, individually-gated PRs/TODOs — see *Deferred work* below):
former PR4 split into *event-driven capture* and *UIA text*; former PR5 *smart enrichment
throttle*.

**Assumption:** single-user, **clean DB / no old data to migrate** for 0.2.0. So there is no
re-embedding pass and no degraded backfill of old frames — but `schema_version` is still bumped.

## PR 2 — Text-Signal Data Model + OCR Spans
**Goal:** Add the schema/types for raw text, content text, and text spans — and make OCR emit the
span geometry the filter needs.

**OCR engine change (the real work):**
- Rewrite `crates/ocr` to walk `Lines().Words()` and emit per-line/word text with `BoundingRect`,
  normalized to `[0,1]`.
- Extend `OcrResult` with spans. WinRT exposes no per-word confidence, so the
  `CONFIDENCE_UNKNOWN` sentinel stays.

**Schema (clean DB, `schema_version` bump):** every per-frame table uses
`REFERENCES frames(id) ON DELETE CASCADE`, matching the existing `ocr_text` / `embeddings`
convention (`crates/store/src/schema.rs`) so retention purges (`storage.retention_days`) don't
orphan rows.
- `frame_text`:
  - `frame_id INTEGER PRIMARY KEY REFERENCES frames(id) ON DELETE CASCADE`, `raw_text TEXT NOT
    NULL`, `content_text TEXT NOT NULL`, `primary_source TEXT NOT NULL` (`ocr` / `uia`),
    `filter_version INTEGER NOT NULL`, `suppressed_count INTEGER NOT NULL`,
    `target_window_title TEXT`, `target_app_hint TEXT`, `created_at INTEGER`.
  - **Interim populator (PR2 lands before PR3's classifier):** PR2 fills `content_text` as a
    **passthrough copy of `raw_text`** (the column is `NOT NULL`). Frames captured in the PR2→PR3
    window keep that unfiltered content_text with no backfill (clean-DB assumption); PR3's filter
    applies to captures from its deploy onward. PR1 records this in `specs/07_KNOWN_GAPS.md`.
- `text_spans`:
  - `frame_id INTEGER REFERENCES frames(id) ON DELETE CASCADE`, `span_index`,
    `PRIMARY KEY (frame_id, span_index)`, `text`, `normalized_text`, `source` (`ocr` / `uia`),
    `role` (`content` / `chrome` / `background` / `system` / `unknown`),
    normalized bbox `x`, `y`, `w`, `h`, `is_searchable`, `suppress_reason`.
- `chrome_text_catalog`:
  - `signature PRIMARY KEY`, `app_hint`, `region_bucket`, `normalized_text`, `seen_count`,
    `first_seen_at`, `last_seen_at`, `suppressed BOOLEAN`.

**FTS decision (pick one in PR2 and document it):**
- Default search → FTS over `content_text` (`frame_text_fts`).
- `include_chrome=true` → also search raw, via either a raw FTS or a role-filtered spans FTS.
- With a clean DB, `frame_text.raw_text` can be the single raw store — no need to keep a legacy
  `ocr_text` table.

**Type changes:**
- Add `TextSource`, `TextRole`, `SuppressReason`.
- Extend `FrameDetail` with `raw_text`, `content_text`, `text_source`, `suppressed_text_count`
  (`crates/traits/src/ipc.rs`).
- Add `include_chrome: bool` (default `false`) to `SearchQuery`.
- Regenerate ts-rs bindings (`git diff --exit-code -- ui/src/bindings` must be clean).

**Acceptance:**
- Default search uses content text FTS + vector.
- Raw text remains viewable in Moment.
- New captures produce spans with normalized bounding boxes.

## PR 3 — Attention-First Text Filtering
**Goal:** Stop toolbars, icons, taskbars, static app nav, and background windows from dominating
retrieval.

Implement a text-filter module.

Inputs: full raw OCR/UIA text; text spans with bounding boxes; foreground/target window rect;
foreground app/window title; capture trigger; the existing chrome catalog.

Classification:
- `system`: taskbar, desktop icons, tray, clock, Start/search bar.
- `background`: visible text outside the target/foreground window.
- `chrome`: menus, tabs, sidebars, toolbars, status bars, repeated app labels.
- `content`: document/editor/browser/terminal/chat/report body text.
- `unknown`: kept only when not obviously static noise.

Default content text:
- Include `content` and useful `unknown` inside the target window.
- Exclude `system`, `background`, `chrome`.
- Include the foreground window title as metadata, not as repeated body text.

Static suppression:
- Normalize spans; build a signature from `app_hint + normalized_text + region_bucket`.
- Mark as chrome after repeated appearances — but **thresholds are configurable in Settings, not
  hardcoded** (the project already hit this anti-pattern with a hardcoded vision batch size).
- Never suppress long, information-rich lines solely because they repeat; never suppress text
  inside `content`/editor roles solely because it repeats.
- `filter_version` is bumpable to recompute the whole catalog.
- Expose a per-app **suppression-rate metric** so over-suppression is observable — false
  suppression is the top risk (silent data loss). `include_chrome` + preserved raw text always
  recover anything wrongly suppressed.

No backfill: the clean-DB assumption means old frames are not retro-filtered. New captures get
span-aware filtering; embeddings run over `content_text` from the first capture.

Test fixture: commit a small **anonymized synthetic** OCR fixture for a deterministic CI classifier
test with golden content-vs-chrome assertions. Do not commit any personal screenshot.

**Acceptance:**
- Default content search no longer ranks frames because of desktop icons or toolbars.
- `include_chrome=true` can still find those raw/static terms.
- Embeddings use `content_text`, not raw full-screen OCR.

## PR 6 — Recall Reports + Ask Shortcuts
**Goal:** Add reports and premade prompts on top of cleaned content text.

**Backend:**
- Add `generate_report(ReportRequest) -> ReportResponse`. Daily = local today; Weekly = trailing
  7 local days; Custom = selected range + optional prompt.
- Use `frame_text.content_text`, never raw full-screen text. Cite frames. No saved-report table.
- **Context strategy (tiered, weak-hardware-safe, no `ctx_size` bump):**
  - Daily / small range → **single pass** over content text. Filtered text is ~150–400 tok/frame,
    so ~20–40 frames fit the pinned 8192 answer context.
  - Weekly / large range → **map-reduce**: batch-summarize frames to fit 8192, then summarize the
    summaries. VRAM-flat; costs more sidecar calls.
  - Make **retrieval depth and reply budget per-request** (remove the hardcoded `ASK_TOP_K = 8` in
    `src-tauri/src/lib.rs`). The budgeter in `crates/inference/src/answer.rs` (`build_messages`)
    already packs frames best-first, drops overflow, and cites only what the model actually read.
  - Leave `sidecar_ctx_size` as the existing power-user / hardware knob — not bumped by default.
  - Framing is honest: "retrieve up to N, summarize what fits" — not a guarantee that all N frames
    are read.

**UI** (resurrecting the old `screensearch` Recall Report-mode UX):
- Recall modes: Search / Ask / Reports.
- Report mode: daily / weekly / custom range → Generate → Markdown render + clickable source-frame
  chips + Copy + `.md` download + a model/tokens footer.
- Ask **premade prompt cards** (genuinely new — none existed in the old UI): e.g. Day Recap,
  Standup Update, Time Breakdown, Top of Mind, AI Habits. Clicking a card fills and submits the
  prompt.
- Scheduled saved reports remain deferred.

**Acceptance:**
- Reports summarize meaningful work/content, not toolbar labels.
- Empty / no-evidence windows produce honest output.
- Raw / app-chrome search remains opt-in.

## Deferred work (0.2.1 — not in 0.2.0)

These were the original PR4/PR5. They are deferred to the 0.2.1 line because they are the
highest-risk, most-invasive parts and are not needed for the retrieval fix. Each becomes its own
gated PR/TODO and is recorded in `specs/07_KNOWN_GAPS.md`.

- **Event-driven capture** (former PR4, part 1): app-switch / focus / click / typing-pause /
  scroll-stop / clipboard / idle triggers. Requires global input + clipboard hooks that do not
  exist today. Prefer `SetWinEventHook` (focus/foreground) and `AddClipboardFormatListener` over
  low-level keyboard hooks; gate behind an opt-in setting and document the privacy posture. It does
  not violate the "no real-time vision" rule (capture ≠ vision), but it is a notable
  privacy/stability/testability shift. **0.2.0 keeps timer/idle capture.** No raw keystrokes or
  clipboard contents are ever stored.
- **UIA text** (former PR4, part 2): UI Automation text from the target window, OCR fallback.
  Greenfield, stateful, thread-affinity-sensitive; needs a capability probe + measured latency
  budget. Own PR.
- **Smart enrichment throttle** (former PR5): pause `vision_tag` / `embed_image` and reduce
  `embed_text` concurrency under sustained CPU/GPU pressure, while capture/OCR/storage never pause.
  Blocked on a CPU/GPU pressure probe that does not exist yet (only a flag-capability probe does,
  `crates/inference/src/flags.rs`). GPU utilization is NVIDIA/NVML-only, so a weak iGPU laptop would
  be CPU-pressure-only regardless — implement with truthful status.
